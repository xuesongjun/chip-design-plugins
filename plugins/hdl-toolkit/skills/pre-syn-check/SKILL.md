# 综合前代码质量检查

对用户指定的 RTL 文件、目录或当前项目执行综合前检查，识别可能导致综合警告或错误的代码模式。

## 执行步骤

1. **确定检查范围**
   - 如果用户提供了路径参数 `$ARGUMENTS`，对该路径执行检查
   - 如果没有提供，询问用户：检查整个 `rtl/` 目录还是某个具体文件

2. **执行检查**

   按严重程度检查以下项目：

   ### ERROR（必须修复）

   | 检查项 | 说明 |
   |--------|------|
   | 多驱动信号 | 同一信号在多个 always 块中被赋值 |
   | 位宽严重不匹配 | 如 `wire [3:0] a; assign a = 32'hFFFFFFFF;` |
   | 未连接的输出端口 | 模块输出未驱动 |
   | 组合逻辑环路 | 输出反馈到自己的输入 |
   | 模块例化端口名错误 | `.port_name(signal)` 中 port_name 不存在 |

   ### WARNING（强烈建议修复）

   | 检查项 | 说明 |
   |--------|------|
   | 未使用的信号 | 声明了但从未使用 |
   | 未使用的端口 | 输入端口未读取或输出端口悬空 |
   | 未覆盖的 case 分支 | 缺少 default 或某些值未处理 |
   | 位宽轻微不匹配 | 如 `wire [7:0] a; assign a = 9'h1FF;` |
   | 敏感列表不完整 | 非 `always @(*)` 的组合逻辑块 |
   | 隐式声明 | 未声明就使用的 wire（依赖默认 net 类型） |
   | 数字字面量无位宽 | 如 `assign a = 5;` 应为 `assign a = 8'd5;` |

   ### INFO（建议优化）

   | 检查项 | 说明 |
   |--------|------|
   | 长信号路径 | 单个 always 块过于复杂，建议拆分 |
   | 魔数 | 直接使用数字常量，建议用 localparam |
   | 模块端口过多 | 超过 50 个端口的模块建议重构 |
   | 嵌套层级过深 | if 嵌套超过 4 层 |

3. **输出报告**

   ```
   === 综合前检查报告 ===

   文件: rtl/bridge.v

   [ERROR] 第 45 行
     信号 'data_out' 在两个 always 块中被赋值（第 45 行和第 67 行）
     综合影响: 综合工具会报多驱动错误
     修复: 合并到一个 always 块，或拆分信号

   [WARNING] 第 78 行
     case 语句缺少 default 分支
     综合影响: 可能推断出 latch
     修复: 添加 default 分支

   [WARNING] 第 102 行
     信号 'temp_data' 声明但未使用
     综合影响: 综合工具会优化掉，但产生 warning
     修复: 删除未用信号

   [INFO] 第 134 行
     直接使用了魔数 256
     建议: 定义为 localparam 提高可维护性

   ---

   总结: 1 ERROR / 2 WARNING / 1 INFO
   ```

4. **后续动作**

   - 如果有 ERROR，强烈建议修复后再综合
   - WARNING 应在 plan.md 中规划修复时间
   - INFO 可在重构时一并处理
   - 询问用户是否需要为某个具体问题给出代码修复建议

## 与 systemverilog-lsp 的关系

如果项目同时安装了 `systemverilog-lsp`，那个插件的 Verible 工具能自动发现并修复部分格式类问题（trailing space、行宽等）。本 skill 关注**语义级**问题（多驱动、位宽、组合环路等），两者互补。

建议工作流：
1. 编辑时由 `systemverilog-lsp` 自动修可自动修复的问题
2. 综合前用 `/hdl-toolkit:pre-syn-check` 检查语义问题
3. 综合时再由综合工具做最终把关
