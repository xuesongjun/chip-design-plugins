# HDL 编码规范检查

对用户指定的 RTL 文件、目录或当前项目执行编码规范检查。

## 执行步骤

1. **确定检查范围**
   - 如果用户提供了路径参数 `$ARGUMENTS`，对该路径执行检查
   - 如果没有提供，询问用户：检查整个 `rtl/` 目录还是某个具体文件

2. **加载规范**
   - 优先参照 `coding-style` skill 中的规范（应已自动加载）
   - 如有项目级 `CLAUDE.md` 中的额外规则，一并应用

3. **执行检查**

   按类别检查以下项目：

   ### 命名规范
   - 时钟信号是否符合 `xxx_clk` / `xxx_gclk`
   - 复位信号是否符合 `xxx_rstn` / `xxx_rstn_sync`
   - 寄存器/组合输出是否符合 `xxx_q` / `xxx_d`
   - 低有效信号是否带 `_n` 后缀
   - 模块名是否与文件名一致
   - 参数/宏是否全大写下划线

   ### 格式
   - 缩进是否为 4 空格（无 tab）
   - `begin` 是否与块同行（如 `always @(...) begin`）
   - 模块顶层结构之间是否空行

   ### 可综合性
   - RTL 中是否出现 `#delay`、`initial`、`force`、`$display` 等仿真语法
   - `if-else` 是否完整覆盖所有分支
   - `case` 是否有 `default`
   - 是否存在潜在 latch 推断

   ### 赋值规则
   - 同一 `always` 块中是否混合阻塞和非阻塞赋值
   - 时序逻辑是否使用非阻塞赋值
   - 组合逻辑是否使用阻塞赋值

   ### FSM
   - 状态机是否为三段式
   - 状态寄存器是否命名为 `state_q` / `state_d`

   ### 复位
   - 是否使用异步复位、同步释放模式
   - 复位条件是否正确（`if (!xxx_rstn)`）

4. **输出报告**

   ```
   === 编码规范检查报告 ===

   文件: rtl/bridge.v

   [命名] 第 12 行
     信号 'clock_a' 不符合命名规范
     建议: 改为 'xxx_clk' 或 'xxx_gclk' 形式

   [格式] 第 34 行
     使用了 tab 缩进
     建议: 改为 4 空格

   [可综合性] 第 56 行
     使用了 #10 延迟语句
     建议: RTL 中禁用，移到 testbench

   [赋值] 第 78 行
     always 块中混合使用了阻塞和非阻塞赋值
     建议: 时序逻辑只用非阻塞，组合逻辑只用阻塞

   ---

   总结: 4 个问题（1 命名 / 1 格式 / 1 可综合性 / 1 赋值）
   ```

5. **后续动作**

   - 询问用户是否要逐个修复
   - 如果是项目历史代码，建议优先修复"可综合性"和"赋值"两类（影响功能正确性）
   - 命名和格式问题可以批量重构，但需注意接口兼容性
